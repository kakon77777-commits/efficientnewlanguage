# EML conformance suite

`docs/EML-LANG-2026-v1.0.md` §11 defines *what* a conforming implementation must do (deterministic
§3 grammar transpilation, §5 semantics, §8 trace envelopes, the §9 round-trip guarantee). This
document is the practical companion: *how to actually check* whether an implementation (this one,
or an alternative) satisfies that contract — externally, without needing to read this repo's
internal `vitest` suite or understand its package layout.

The suite has two layers, covering the two things §11 requires: exact deterministic transpilation,
and correct execution.

## Layer 1 — grammar-mapping conformance (`tests/fixtures/`)

29 minimal fixture pairs, one per grammar construct from Phase 0 through Phase 7 (arithmetic, Σ,
matrix/transpose, conditionals, augmented assignment, ranges, cold functions, `if`/`elif`/`else`,
`while`, `for`/`break`/`continue`, dict/set literals, subscript assignment, `import`, `try`/`except`/
`finally`/`raise`, `class`). Each `NN_name.eml` has a committed `NN_name.expected.py` — the exact
Python text §4/§5's overlay-semantics table says that construct MUST expand to.

**This is a text-exact check, not a "produces equivalent output" check** — per §2.1's "ASCII
canonical form is normative" stance and §11's determinism requirement, two conforming
implementations of the same EML source MUST emit byte-identical Python, not merely
behaviorally-equivalent Python. This is the strongest, most falsifiable form of conformance the
language's own design affords.

**These fixtures are deliberately snippets, not standalone programs.** Several (e.g. `04_transpose`,
`05_condition`, `06_bind`, `10_range`, `12_matrix`, `13_call`) reference names that are never bound
(`m`, `x`, `f`, `data`, ...) — they exist to pin down one construct's expansion (Appendix B's "14
cases" and their Phase 6/7 successors), not to run standalone. Don't `python -c` the matched output
of an arbitrary fixture and expect it to execute cleanly; that isn't what this layer checks or
claims to check. Execution-truth is Layer 2's job.

Run it directly, no `vitest`/dev-deps required beyond the CLI itself:

```bash
pnpm eml test                    # tests/fixtures/*.eml -> exact Python-text match
pnpm eml test --dir path/to/dir  # point at an alternative fixture set (e.g. a fork's own)
```

Exit code `0` = every fixture passed; `1` = at least one failed — scriptable for CI. `pnpm eml test`
is also what `tests/golden.test.ts` checks internally (same fixtures, same comparison), so this is
not a parallel or weaker guarantee — it is the same one, exposed as a standalone command an external
verifier can point at their own implementation's output.

## Layer 2 — execution-truth conformance (`examples/`)

The fixture pairs above prove correct *mapping*; they don't prove the mapped Python, once *run*,
produces the program's intended output on real, non-trivial control flow (loops, recursion,
exceptions, classes). `examples/` — a smaller set of complete, runnable programs, including two
genuine third-party ports (`examples/mvp-tic-tac-toe/`, `examples/mvp-number-guessing-game/`) — is
the execution-truth layer: each carries a committed `.trace.jsonl` `phosphor-jsonl-v1` golden
generated by this repo's own execution-truth interpreter (`@eml/interp`), which is itself gated by
an `eml:equiv` check against a **real Python process** (§9, §11's "correct execution" half of
conformance).

```bash
pnpm eml trace examples/mvp-tic-tac-toe/tic_tac_toe.eml --run   # -> eml:equiv ok:true, or a failing exit code
```

`--run` compiles the example, executes it under both this repo's interpreter and a real `python`,
and emits an `eml:equiv` event comparing the two stdouts — the strongest available proof that a
given `.eml` file's *documented, intended behavior* (not just its transpiled text) is correct.

## Known scope boundary (documented, not silently missing)

The two layers are deliberately separate mechanisms, not a gap: Layer 1's snippet fixtures prove
*mapping* (this construct expands to exactly this Python), Layer 2's programs prove *execution*
(this complete program, once run, produces exactly this output). Neither layer subsumes the other —
a snippet fixture can't run standalone (see above), and a full example's trace golden doesn't pin
down each individual construct's expansion in isolation the way a fixture does. Unifying them into
one mechanism (e.g. giving every fixture a runnable, self-contained variant) is future work, not
attempted this round.

## For an alternative/forked implementation claiming conformance

1. Point Layer 1 at your implementation's transpile output for `tests/fixtures/*.eml` — every
   `.expected.py` MUST match byte-for-byte (modulo the fixture's own EOL normalization).
2. Run Layer 2's examples through your implementation end to end — real Python execution of your
   transpiled output MUST match the committed `.trace.jsonl`'s `eml:output` sequence.
3. See `docs/EML-LANG-2026-v1.0.md` §11 for the exact normative MUST/MUST NOT clauses this suite is
   checking, and Appendix A for the frozen diagnostic-code contract (a producer's diagnostic codes
   are part of the conformance surface — messages may differ, codes MUST NOT change meaning).
